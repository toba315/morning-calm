## 2026-06-27 GOOD-29: ニュース記事の取得元選定と収集システム導入

### 背景

GOOD-29では、Morning Calmで扱う「事実ベースの明るいニュース」をどこから、どの条件で取得するかを決めた。
ニュース記事には著作権があり、他社ニュース本文を無断でAIに入力して、元記事を読まなくても内容が分かる要約をアプリ内に掲載するのは避ける方針にした。

### 決めたこと

- 他社ニュース記事本文のスクレイピングや要約配信はしない。
- NewsAPI、Google News RSS、一般ニュースサイトRSSは、競合サービス構築や本文利用の規約リスクがあるため本番の主ソースにしない。
- 基本方針は「公知の事実だけをベースに、AIまたはテンプレートで完全オリジナル文を作る」。
- 取得対象は、公式RSS、公式新着ページ、政府・自治体・研究機関・施設公式ページ、CCライセンスで扱えるソースに限定する。
- 記事本文は保存しない。RSSのdescriptionも本文生成には使わない。
- アプリ内には出典リンク、ライセンス名、帰属表記を保持・表示できるようにする。
- 水族館・動物園・博物館・科学館などの公式ニュースは、明るい一次情報として相性が良いので候補に入れる。
- WikinewsはCCライセンスの候補として採用可能。ただし現時点では本文を流用せず、タイトル・URL・日付などのメタ情報をもとに独自文を生成する。

### 導入したもの

- `npm run news:fetch`
  - ニュース収集を実行するコマンド。
  - `scripts/fetch-news.mjs` を起動する。

- `scripts/fetch-news.mjs`
  - 公式HTML一覧、RSS/Atom、MediaWiki APIからニュースのメタ情報を取得する。
  - 取得する主な情報は、タイトル、URL、日付、発表元、カテゴリ、ライセンス情報。
  - 文字コードの違いに対応するため、レスポンスのcharsetやHTMLのmeta charsetを見てデコードする。
  - 各ソースは10秒でタイムアウトし、失敗しても他のソースの処理を続ける。
  - ネガティブ・運用告知系のタイトルを除外し、明るい話題を優先する簡易スコアリングを入れた。

- `scripts/news-sources.json`
  - 取得元の定義ファイル。
  - 現在の主な候補:
    - すみだ水族館
    - 京都水族館
    - サンシャイン水族館
    - Wikinews
    - 理化学研究所
    - 経済産業省

- `news-data.js`
  - `npm run news:fetch` の生成物。
  - `window.MORNING_CALM_NEWS` として画面側から読み込む。

- `app.js`
  - 固定デモ記事ではなく、`news-data.js` の生成データを優先して表示するように変更。
  - 詳細モーダルで公式発表リンク、確認ファクト、ライセンスリンク、帰属情報を表示できるようにした。

- `index.html`
  - `news-data.js` を `app.js` より前に読み込むように変更。
  - 文字化けしていた主要UI文言を整理。

- `docs/news-ingestion.md`
  - ニュース取得の運用方針、ソース追加方法、Wikinewsの扱いを記録。

### 実行時に分かったこと

- 日本語サイトの一部はShift_JIS系で配信されており、UTF-8固定で読むと文字化けする。charset判定を入れて解消した。
- 水族館サイトは取得しやすいが、「台風接近に伴う営業」「休館」「営業時間」などの運用告知が混ざる。明るいニュースだけにするには除外語の調整が必要。
- WikinewsはRSS/AtomよりMediaWiki APIで取得するほうが安定した。
- 今回の最終生成5件は、スコアリングの結果として理化学研究所の公式発表が中心になった。Wikinewsは候補プールには入っているが、上位5件には入らなかった。

### 残っている課題

- 明るいニュース判定はまだ簡易ルールなので、GOOD-14/GOOD-15のAIポジティブフィルターと連携して精度を上げる。
- 水族館・動物園・博物館系は、サイトごとにHTML構造が違うため、必要に応じて個別パーサーを追加する。
- 経済産業省はタイムアウトしたため、RSSや別エンドポイントがないか再調査する。
- 現在はテンプレート生成で、AIによる完全オリジナル記事生成は未接続。後続でAI生成APIとつなぐ。
- 生成データ `news-data.js` は手元実行で更新されるため、本番では定期実行・キャッシュ・失敗時の前回データ保持を設計する。
